# SF-ABS-0001 - Domain-Remedy Absorber Check

## Boundary

This absorber check uses only the already admitted `SF-0001` and `SF-0002`
records. It does not add source material, create an accepted topology schema,
make an aviation or spaceflight recommendation, or authorize any intervention.

The test asks whether the provisional review-chain fields collapse into a
weaker explanation:

> these are just domain-specific technical failures whose remedies belong in
> aviation and spaceflight safety practice.

If that absorber explains everything this repository needs to preserve, the
provisional topology fields should be killed or demoted.

## Absorber Fit

The absorber is partly valid:

- each record has domain-specific technical substance outside this repository's
  authority;
- the right remedies, if any, belong to domain authorities and affected
  owner-governance paths;
- this repository must not treat a generic field list as a substitute for
  engineering, certification, or operational judgment.

That valid portion narrows the topology work. It prevents this repository from
promoting an ontology that claims to explain accidents or prescribe fixes.

## Why It Does Not Kill The Pattern

The absorber does not explain the review-chain residue preserved by both
records:

| Provisional field | Residue not absorbed by domain remedy |
|---|---|
| assumption identifier and source revision | The old assumption must remain tied to the evidence context being reused. |
| accountable review owner | Domain remedy does not by itself show who owned the contradiction before harm. |
| affected-party standing | The person or system carrying risk can disappear behind technical process. |
| change or observation event | The review burden changes when new conditions or evidence arrive. |
| validation burden | The old assumption cannot remain valid merely because prior practice accepted it. |
| operating or observation environment | The context in which evidence is seen or missed remains material. |
| delegation path and visibility gap | Distributed review can hide contradiction even when domain expertise exists. |
| correction route and stop condition | A valid domain remedy still needs a visible route for unresolved evidence. |

The useful pattern is therefore not "generic accident cause." It is narrower:
source-backed assumption reuse under changed conditions needs a visible owner,
validation burden, affected-party standing, and correction route before old
assumptions are treated as still adequate.

## Field Disposition

Current disposition: keep the field set provisional and narrower.

Do not promote `hazard_history` or `observation_request_path` yet. They remain
candidate pressures because this absorber check did not add a new domain
record.

## Next Test

The next useful test is still a third source-backed record from a non-aerospace
domain. It should try to falsify the review-chain pattern by showing a systemic
failure where assumption reuse, validation burden, or owner-visible correction
is not central.
